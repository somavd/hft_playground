Distributed file downloader — split, download in parallel, reassemble.

Components

coordinator binary

desktop-test-harness/coordinator.cpp

  • Listens for HELLO/LEAVE messages
  • Maintains peer address map
  • Runs round-robin chunk assignment
  • Broadcasts CHUNK_ASSIGNMENT to all peers on cluster change

peer_node binary

desktop-test-harness/peer_node.cpp

  • Sends HELLO to coordinator on start
  • Waits for CHUNK_ASSIGNMENT
  • Downloads assigned chunks via HTTP range
  • Reassembles into output file
  • Sends LEAVE on exit

ChunkManager core

core/chunk_manager.h/cpp

  • Splits file into 1 MB chunks
  • Maintains peer registry + chunk bitmaps
  • Round-robin assignment algorithm
  • Tracks download status per chunk

PeerDiscovery core

core/peer_discovery.h/cpp

  • TCP server + client
  • 3 message types: HELLO, LEAVE, CHUNK_ASSIGNMENT
  • Network byte order (htonl/ntohl)
  • Callback-based message handling

HttpRangeFetch core

core/http_range_fetch.h/cpp

  • libcurl-based HTTP GET with Range header
  • Writes directly to file
  • Checks Accept-Ranges support

Hasher + Reassembler core

core/hasher.h/cpp & core/reassembler.h/cpp

  • SHA-256 via OpenSSL
  • Offset-based reassembly (ftruncate + seekp)

Wire Protocol

All TCP. All multi-byte fields in network byte order (big-endian).

MessageIDDirectionFormatSize
HELLO1peer → coord[type:1][node_id:4][port:2]7 bytes
LEAVE2peer → coord[type:1][node_id:4]5 bytes
CHUNK_ASSIGNMENT3coord → peer[type:1][count:4][chunk_id:4 × N]5 + 4N bytes

Control Flow

1. Startup

1
Coordinator starts, binds TCP on port 5001
2
Peer starts, creates node_N/ directory, initializes ChunkManager (computes chunk list from file size)
3
Peer adds self to local peer registry, starts TCP listener
4
Peer sends HELLO to coordinator (node_id + listen_port)
5
Coordinator receives HELLO → adds peer → runs round-robin → sends CHUNK_ASSIGNMENT to all peers
6
Peer receives CHUNK_ASSIGNMENT → updates local bitmap → begins downloading

2. Download

1
For each assigned chunk ID, compute byte range: [id × 1MB, min((id+1) × 1MB - 1, file_size - 1)]
2
HTTP GET with Range: bytes=start-end → save to node_N/chunk_ID.bin
3
SHA-256 hash the chunk file, mark as downloaded
4
Log progress every 10 chunks: [NODE 1] progress: 10/26 chunks downloaded

3. Reassembly

1
Pre-allocate output file to file_size bytes via ftruncate
2
For each downloaded chunk: seekp(offset) → write(data)
3
SHA-256 hash final file, print result

4. Shutdown

1
Peer sends LEAVE to coordinator
2
Coordinator removes peer, reassigns remaining, broadcasts new assignments

Round-Robin Assignment

Chunks: [0] [1] [2] [3] [4] [5] [6] ... [50]
Peers:   P1  P2  P1  P2  P1  P2  P1  ...  P1

Result: P1 gets 26 chunks (even IDs)
        P2 gets 25 chunks (odd IDs)

When a new peer joins or leaves, all bitmaps are cleared and reassigned. The coordinator then sends the new assignment to every peer.

Key Code Paths

coordinator.cpp — onPeerJoin

net.onPeerJoin([&](uint32_t id, const std::string& ip, uint16_t port) {
    addrs[id] = {ip, port};              // Store peer address
    cm.addPeer(p);                       // Register in ChunkManager
    cm.assignChunksRoundRobin();         // Recompute assignment
    broadcastAssignments();              // Send to ALL peers
});

peer_node.cpp — onChunkAssign

net.onChunkAssign([&](const std::vector<uint32_t>& ids) {
    auto it = peers.find(node_id);       // Find self in registry
    it->second.chunk_bitmap.reset();     // Clear old assignment
    for (uint32_t id : ids)
        it->second.chunk_bitmap.set(id); // Set new chunks
});

ChunkManager — Round Robin

for (size_t i = 0; i < chunks.size(); ++i) {
    auto it = peers.begin();
    std::advance(it, i % num_peers);     // Cycle through peers
    it->second.chunk_bitmap.set(i);      // Assign chunk i
}

Limitations (MVP)

  • No P2P chunk exchange — Peers download from HTTP server only, no peer-to-peer transfer yet
  • Snapshot assignment — Mid-download reassignment doesn't cancel in-progress chunks
  • No resume — Crash requires full re-download of assigned chunks
  • No master hash manifest — Each peer hashes its own chunks independently
